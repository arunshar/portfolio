Abstract

I built Harbormaster to honestly close resume gaps I named in a Supabase Multigres cover note: change-data-capture, streaming, production distributed systems, and MLOps observability. It is a real-time maritime anomaly-detection platform on AWS, live AIS vessel traffic through Kinesis and Managed Flink into a deterministic scoring front door, a Postgres change-data-capture pipeline keeping an online watchlist fresh, and a SageMaker-served model with a full promotion pipeline and a drift-to-HITL-to-RL flywheel behind it. Three layers of adversarial review sit on top of the code: two internal reviews that fixed 26 findings between them, an external audit against AWS's own canonical examples that caught a real, dollar-denominated bug before it ever ran, and live AWS runs that found what only reality could. All of it runs under a $75-a-month hard cost cap enforced by an IAM-deny policy, not a spreadsheet estimate.

This page covers what got built, phase by phase, the promotion pipeline and drift flywheel at the center of it, the adversarial-review discipline that differentiates it from a demo, and what it honestly does not prove.

1. Why this exists, honestly

At ESRI, in the summer of 2023, I worked on a real maritime anomaly detector and the AWS MLOps around it, on a real team with real clients. That work is ESRI's. Harbormaster is my personal extension of the same problem, built entirely after that engagement, on public data, never merged with ESRI code or infrastructure.

Closes: change-data-capture, streaming, production distributed systems, MLOps and infrastructure-as-code, observability.
Does not close: a sharded query router, a consensus implementation. That is Vitess and Multigres territory, and Harbormaster deliberately consumes managed Postgres rather than re-implementing it.

2. Guardrails before anything

Phase 0, the very first thing built, was the cost cap itself: a $75 hard budget whose breach action attaches an IAM-deny policy directly to the platform role, a soft budget alerting at $5, $15, and $25, cost-anomaly detection, and a nightly teardown Lambda that sweeps orphaned resources. This came before any pipeline, any model, any line of scoring logic. A deny policy stops new spend, it does not stop spend already in flight, so the cap needed the teardown Lambda alongside it, not the deny policy alone, a lesson the platform names as its own first anticipated war story.

3. Architecture: two planes, one code path

Training happens on MSI, free H100s, entirely off the cloud. Serving happens on AWS. Local fakes and injected I/O prove the same code path the AWS demo windows exercise for real, so nothing has to be taken on faith. Kinesis carries the telemetry stream; Kafka, behind Debezium, carries the change-data-capture stream, two different reliability shapes, two different brokers. There is no standing load balancer and no Kubernetes control plane in the phases built so far: an EKS control plane alone runs about $73 a month flat, whether or not anything is running on it, and would eat the entire cap, so the serving front door runs on ECS Fargate instead, a decision a later, already-signed-off phase revisits on its own terms.

Two planes: MSI trains, AWS serves. Solid arrows are the live telemetry path (Kinesis to Flink to the scoring service to HITL); the lower row is the change-data-capture path keeping the online watchlist fresh; the dashed arrows are the one-way checkpoint manifest and the read-through watchlist lookup the scorer uses on every request.

4. The build in five phases

Phase 0, deployed, live since 2026-07-03. VPC, state stores, the FinOps guardrails above. The only always-on plane; everything else is a bounded demo window, toggled on for a session and torn down before the window closes, never left running between sessions.

Phase 1, AWS showcase run live 2026-07-04. A replay ingestor into Kinesis, Managed Flink computing per-vessel features and a cheap physics gate, a deterministic ECS Fargate scoring front door with the LLM planner stripped out entirely, a Streamlit human-in-the-loop console. The platform's first-ever live run against real infrastructure found and fixed five real bugs, ranging from a Python worker subprocess that does not inherit its driver's environment to a replay fixture's historical timestamps expiring a database record the instant it landed. Once fixed, a planted anomaly reached the review queue end to end, confirmed live.

Phase 2, local-stack accepted 2026-07-03; AWS showcase not yet run. The resume-critical piece: Postgres logical decoding through Debezium into an idempotent, LSN-guarded consumer, keeping a DynamoDB and Redis online store fresh, with an append-only audit trail and replication-slot lag alerting. All five acceptance criteria pass in one 33.5-second run against the local stack, and a same-day adversarial review found and fixed seventeen findings, including a real, silent Phase 1 defect it caught by accident. I say plainly that the AWS demonstration of this specific pipeline has not run yet, and no claim on this page should read otherwise.

Phase 3, AWS showcase run live 2026-07-04, then torn down clean. A transient EMR Spark job backfills historical AIS into an Iceberg lake; a SageMaker async endpoint serves a model behind a real scale-to-zero mechanism, verified live in both directions, a genuinely zero-instance endpoint waking to serving in about 30 seconds; a full promotion pipeline, holdout gate through weighted canary with burn-rate auto-rollback, ran end to end against the live endpoint and promoted a candidate successfully, matching a pinned expected transition sequence exactly. Afterward, every resource, thirteen of them, was torn down in one clean pass and verified gone.

Phase 4, code-complete on its integration branch, mergeable. The drift-to-HITL-to-RL flywheel: three drift detectors answering three different questions (population shift, calibration, and a proxied concept-drift signal cross-validated against real human disagreement), a preference-triple pipeline for reinforcement-learning fine-tuning candidates, and a reward-hacking probe I red-teamed myself and found a real gap in the same day it was built, then fixed with the smallest possible change before the day was over.

Phase 5, authored and signed off; the build is a later sprint. A multi-tenant EKS front door with KEDA-based scale-to-zero, per-tenant row-level-security isolation and SLOs, a Bedrock explanation layer that narrates already-computed reasons and never touches the score itself, and a labeled PPO route-optimization stretch service, never counted toward this platform's core resume-gap claims. Signing off on the plan surfaced three real gaps in the platform's own prior assumptions before any of this phase's code exists, the kind of finding a plan review is supposed to produce.

5. The CDC pipeline: the gap that mattered

The whole reason Harbormaster exists is a line I wrote in a Supabase Multigres cover note admitting I had not built change-data-capture in production. This pipeline closes it. Postgres emits a durable, ordered log through logical replication; Debezium captures it; a consumer applies every change through a monotonic LSN guard enforced by a DynamoDB conditional write, so replaying the entire topic from the beginning produces zero duplicate state even though every redelivery is still recorded, on purpose, in an append-only audit table. Insert-to-online latency measures at 0.57 seconds against a 5-second target. Two live drills grounded the design: a stalled consumer's replication-slot lag climbing from zero to 74 million bytes before the alert fires and the slot drains, and a consumer restart under a hostile schedule proving the guarded path converges to byte-identical state where an unguarded path visibly regresses.

6. The promotion pipeline and the flywheel

Every model candidate walks through one door: a holdout gate reusing ported, cross-checked evaluation code, then a shadow comparison against real online traffic, then a canary ramp at 5, 25, 50, and 100 percent, with a burn-rate breach at any weight triggering a full, immediate revert. This is modeled as a saga, not an ad hoc script: each promotion step has an explicit compensating action, so a bad canary weight has a tested, automatic undo rather than an incident-time scramble. Two drills prove the layers actually catch what they claim: a candidate that passed the holdout gate cleanly (AUC 1.0) failed shadow with a real divergence the offline check structurally could not see, because the offline export and the online path had drifted onto different feature encodings; and a candidate that passed both holdout and shadow regressed only once real canary traffic reached it, reverting cleanly at weight 25 and never advancing further. On top of the promotion pipeline sits the drift flywheel: input drift, calibration drift, and a proxied concept-drift signal each answer a different question, cross-validated against each other so a rising population-shape signal alone never triggers retraining on its own, only a rising rate of real operator disagreement does. A reward-hacking probe guards the one path where a fine-tuned candidate could game its own reward function, and I red-teamed that probe myself and found a real gap, a candidate that could win on reward while making its physics violations more severe without ever tripping the original rate-only check. Fixed the same day, with the smallest possible change, and the predicted blast radius held exactly: one existing test's expected outcome flipped, correctly, and nothing else moved.

7. Red-teaming my own platform

Four layers of review, not one. Internally, a Phase 1 review confirmed nine findings and a Phase 2 review confirmed seventeen, including a real, live, silent defect: the serving task definition on AWS had never actually injected the database credential, so the human-review queue had been quietly falling back to an in-memory stub the entire time, caught by accident while reviewing something else. A separate Phase 4 review confirmed the reward-hacking probe's own gap. Externally, an independent audit ran the Phase 3 SageMaker Terraform line by line against AWS's own canonical examples and found nineteen distinct findings, two of them fix-before-demo severity, seven hardening, six decision-record-only, and four that were checked and held without change, including a scale-to-zero design fuller than AWS's own walkthrough notebook. The headline: a target-tracking metric specification was missing one required dimension, meaning the platform's scale-to-zero mechanism would never have fired and an idle GPU instance would have run twenty-four hours a day at roughly 530 dollars a month, silently, against a 75-dollar cap. Both fix-before-demo findings are fixed now, at real commits, verified live, not just planned. Verifying that fix live then surfaced a second, sibling bug on a different alarm the audit itself never checked, an extraneous dimension on the opposite alarm that quietly kept the wake-from-zero path from ever firing, the platform's clearest argument that a review which catches a bug class does not certify every other instance of it is safe. Live AWS runs found what only reality could: a Python UDF worker subprocess that does not inherit its driver's environment, a runtime that turned out to be Python 3.9 rather than the development machine's 3.12, a TTL sweeper deleting historical replay data within seconds of it landing, a Docker build producing a manifest format the deployment target flatly rejects. None of these showed up until the pipeline actually ran against real infrastructure, which is the whole argument for running it rather than trusting a clean plan and a passing test suite.

8. What this does not prove

  • No sharded query router and no consensus implementation. That is Vitess and Multigres territory, and I say so before anyone asks.
  • The SageMaker endpoint serves a labeled demo stand-in; there is no MSI-trained checkpoint behind it yet.
  • The canary weight ramp is currently traffic-inert, a single-variant endpoint, and the shadow comparison self-paired against the same batch. A challenger variant is the honest next step before that story is fully load-bearing.
  • Phase 2's AWS demonstration of the CDC pipeline has never run; the pipeline is proven end to end on a local stack only.
  • The real reinforcement-learning fine-tune has not run; the flywheel as built ends at a validated preference-data pipeline.
  • Phase 5 is a signed-off plan, not a build. Nothing beyond the doc exists yet.
  • Restricted figures from my ESRI internship never appear here as a corroborated Harbormaster result; where they matter, I describe them at architecture granularity only.
  • Full-scale historical data, roughly 600 gigabytes of AIS history, is costed and designed but has only run at fixture scale so far.
