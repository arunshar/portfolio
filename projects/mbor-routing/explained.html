MBOR (Multi-Level Bi-Objective Routing) — Explained

One line: real-time electric-vehicle route planning that returns the whole Pareto-optimal set of routes trading off energy against travel time, made interactive by precomputing and reusing the Pareto structure over a region-partitioned graph instead of recomputing it per query.

Honest status up front. MBOR is preliminary research (algorithm design plus a reported timing result), developed with Prof. Shashi Shekhar and tied to Task 4 of his NSF proposal on multi-objective path selection. The reported result, more than 10x faster online than prior methods on real-world road-network data, comes from that preliminary work; it is not yet a public, reproducible benchmark, and unlike the other portfolio projects (GeoSAM-3D, MapFix, DarkVesselNet) there is no public code repository, test suite, or live demo yet. The paper is in preparation. In short: "the design and the data structures are mine; the 10x is a preliminary result I would reproduce as a public benchmark before claiming it as final."

1. What it is and the gap it fills

An electric vehicle does not have one best route, it has a trade-off curve. The fastest route can drain the battery; the most energy-efficient can be slow. The correct object to return is the Pareto-optimal set: every route for which you cannot improve travel time without spending more energy, and vice versa. The driver (or the fleet policy) then picks a point on that curve.

The gap: computing the Pareto set is expensive. Bi-objective shortest path is NP-hard in the worst case, the frontier can contain exponentially many non-dominated paths, and recomputing it per query over a city-scale graph is far too slow for real-time use. Single-objective routing solved its scale problem with preprocess-once, query-fast hierarchies (contraction hierarchies, hub labelling, Shekhar's boundary multigraph). MBOR lifts that bet to two objectives.

2. Architecture and data flow

 PREPROCESS (once):
   road network ─► partition into regions ─► boundary multigraph (BMR)
                                              │  (Pareto inter-region trade-offs as multi-edges)
                                              ▼
                                   multi-level encoded Pareto frontier view
                                              │  (compact non-dominated sets, multiple granularities)
 QUERY (real-time):
   origin/destination + driver trade-off ─► 2D cost-interval + multi-edge pruning
                                              │  (drop dominated fragments early; reuse precomputed frontier)
                                              ▼
                                   complete Pareto-optimal route set

3. The three data structures (the contribution)

Boundary multigraph (BMR). Partition the road network into regions; for each pair of boundary nodes, summarize the Pareto-optimal inter-region trade-offs as parallel (multi) edges. A query then hops region-to-region over this small graph instead of expanding the full network, the hierarchical-acceleration idea (Shekhar's boundary multigraph) generalized from one cost to a two-cost frontier.

Multi-level encoded Pareto frontier view. Store the non-dominated (energy, time) cost sets compactly and at multiple levels of granularity, so the precomputed frontier can be retrieved and stitched at query time rather than re-derived. This is what turns "recompute the exponential frontier" into "look it up and combine."

2D cost-interval pruning with multi-edge pruning. At query time, bound each partial path by a 2D (energy, time) cost interval and discard fragments that are already dominated, and collapse redundant parallel edges. The search therefore never enumerates the full frontier; it prunes aggressively against the precomputed bounds.

4. The core idea, from scratch

Domination. Route A dominates route B if A is no worse on both energy and time and strictly better on at least one. The Pareto frontier is the set of non-dominated routes. Bi-objective label-setting (Martins', NAMOAdr, BOA) computes it by keeping, at each node, a set of non-dominated cost labels, the difficulty is that this set can blow up.

Why preprocessing wins. Most of the work in answering many queries is repeated: the inter-region trade-offs do not depend on the specific origin/destination inside far-away regions. Precompute those once into the boundary multigraph and the frontier view, then a query only stitches boundary-to-boundary frontiers and prunes locally. That is the same reason contraction hierarchies answer single-objective city queries in microseconds, applied to the two-cost case.

Energy as a path-dependent cost. EV energy is not a fixed per-edge number, it depends on speed, mass, and gradient history. The proposal's Pgraph (path-dependent) representation lifts costs from edges to nodes that stand for entire path fragments, so an energy model that depends on the prior trajectory can be expressed exactly. MBOR's frontier machinery is what makes that richer cost tractable.

6. What I'd do next

  1. Release a runnable reference implementation on a public OSM-derived graph (a metro road network) with a simple energy model.
  2. Report a reproducible benchmark: query latency and Pareto-set completeness against NAMOAdr / BOA baselines, the same "make one thing real and measured" bar the other projects meet.
  3. Add the Pgraph path-dependent energy cost end to end and show it changes the frontier vs an edge-additive energy model.
  4. An interactive demo: drag the energy-vs-time preference and watch the chosen route move along the frontier.
