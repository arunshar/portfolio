Towards Pareto-optimality with Multi-level Bi-objective Routing

Abstract

Given an origin, a destination, and a directed graph in which each edge carries a pair of non-negative costs, the bi-objective routing (BOR) problem asks for the complete set of all Pareto-optimal paths, the Pareto frontier of non-dominated routes. This matters whenever a single "best" route does not exist: an electric vehicle trades travel time against energy, and different drivers sit at different points on the curve. BOR is hard because the Pareto-optimal set can be exponentially large, which makes compute-on-demand search too slow for interactive use, while full all-pair precomputation is infeasible to store. We propose Multi-level Bi-Objective Routing (MBOR), built on three ideas: a boundary multigraph representation, a Multi-level Encoded Pareto Frontier View (MEPFV), and two-dimensional cost-interval based pruning. On real road-network data the method runs more than 10× faster online than exact compute-on-demand baselines.

This page presents the full method (problem, the boundary multigraph, the MEPFV encoding, the precompute and online algorithms, the 2D cost-interval pruning, and the correctness guarantee), the paper's experiments, and an independent from-scratch Rust reproduction that reproduces the speedup on the open DIMACS Bay Area network with every answer exactness-checked. All original figures are in the full PDF.

1. The problem: bi-objective routing

Single-objective shortest path (Dijkstra, A*, contraction hierarchies) returns one route in microseconds. The moment two costs cannot be collapsed into one number, the single route is the wrong answer: you want the whole non-dominated frontier. Formally, a path p dominates p' if it is no worse on both costs and strictly better on at least one; the answer is every path that nobody dominates. Applications include eco-routing (time vs energy), collaborative path selection among stakeholders with different objectives, and skyline path queries.

Figure 1 (redrawn). Example network with two costs per edge. The Pareto frontier from n0 to n7 has two non-dominated routes: [n0,n3,n5,n7] = (10,17) and [n0,n1,n3,n5,n7] = (11,16). A route like [n0,n2,n4,n6,n7] = (18,20) is dominated.

2. Problem formulation

Spatial network. A directed graph G = (N, E, c) where each edge has a cost pair c: E → ℝ+ × ℝ+, written (c1, c2).
Path cost. A path's cost is the component-wise sum of its edge costs.
Dominance. For paths p, p', p ≺ p' (p dominates p') iff c1(p) ≤ c1(p'), c2(p) ≤ c2(p'), and c(p) ≠ c(p').
Pareto-optimal set / frontier. POG(o,d) is the set of origin-destination paths that no path dominates; POCG(o,d) is their cost set.
Minkowski sum. For cost sets A, B: A ⊕ B = { a + b : a ∈ A, b ∈ B }; combining sub-path frontiers uses this sum followed by a dominance filter.

BOR is NP-hard and the number of Pareto-optimal paths can be exponential in the worst case, which is exactly why neither pure compute-on-demand search nor full all-pair precomputation scales, and why MBOR precomputes only between region boundaries.

3. Approach: three ideas

Boundary multigraph

Partition G into fragments; the nodes on cross-fragment edges are boundary nodes. A small graph over only the boundary nodes carries the inter-region trade-offs as multi-edges (whole Pareto cost-sets), because two boundary nodes can be connected by several incomparable routes.

MEPFV

The Multi-level Encoded Pareto Frontier View stores, compactly, the within-fragment Pareto sets (FPPV) and the boundary-to-boundary Pareto sets over the boundary multigraph (BPPV), so a query stitches a few precomputed frontiers instead of re-deriving them.

2D cost-interval pruning

In the advanced variant, each candidate boundary-node pair gets a two-dimensional cost-interval; pairs whose interval is dominated cannot contribute to the frontier and are skipped before the expensive Minkowski combination.

4. The boundary multigraph

Partition G into fragments S = {S1,...,Sf} (min-cut, e.g. KaHIP). A node is a boundary node if it has a neighbor in another fragment. The boundary multigraph Gb = (Nb, Eb, Cb) has two kinds of edges:

  • Local multi-edges: for two boundary nodes in the same fragment, each within-fragment Pareto-optimal route between them becomes a parallel edge, and the edge carries the whole Pareto cost set. An ordinary boundary graph (one edge per pair) would discard valid trade-offs, so a multigraph is necessary, not cosmetic.
  • Boundary edges: each original cross-fragment edge is kept, connecting two boundary nodes in different fragments with its single cost.
Figures 2–3 (redrawn). Left: the network split into fragments; green nodes are boundary nodes. Right: the boundary multigraph over only the boundary nodes, with local multi-edges (within-fragment Pareto sets) and boundary edges (original cross-fragment edges).

5. The MEPFV encoding

The Multi-level Encoded Pareto Frontier View has two parts:

  • FPPV (Fragment Pareto-optimal Path View): per fragment, the Pareto sets from each node to each boundary node and back, stored as tuples [destination, next-hop, c1, c2].
  • BPPV (Boundary Pareto-optimal Path View): the Pareto sets between boundary-node pairs over the boundary multigraph, stored as [destination, fragment-of-next-hop, next-hop, c1, c2].
The MBOR pipeline: build the MEPFV once (partition → FPPV → boundary multigraph → BPPV); answer each query by combining FPPV ⊕ BPPV ⊕ FPPV, with MBOR-Adv pruning dominated boundary-node pairs.

6. Algorithms

Precompute (MBOR-Basic, Alg 1–2)

Identify boundary nodes from cross-fragment edges; encode each fragment's Pareto path view (FPPV) by running a bi-objective label-setting search from every node (Alg 2, paretoOptEncoding); build the boundary multigraph by inserting local multi-edges (within-fragment boundary-to-boundary Pareto sets) and boundary edges; finally encode the BPPV by running the same label-setting over the boundary multigraph.

paretoOptEncoding(G):                      # Algorithm 2
  for each start node s (in parallel):
    PQ ← {label(s, cost=(0,0))}            # lexicographic (c1, c2) order
    while PQ not empty:
      l ← PQ.pop_min()
      if c2(l) ≥ c2_min[node(l)]: continue  # truncated dominance
      c2_min[node(l)] ← c2(l)
      for each (edge / multi-edge cost) out of node(l):
        l' ← extend l; if not dominated: PQ.push(l')
    store the Pareto path view from s

Online retrieval (MBOR-Basic, Alg 3)

For a query (o,d), form a superset of the frontier by combining, over boundary-node pairs (oBN, dBN), the precomputed pieces with the Minkowski sum, then filter by dominance:

PO_sup ← ∅                                 # Algorithm 3
for oBN in Boundary(Frag(o)):
  for dBN in Boundary(Frag(d)):
    PO_sup ← PO_sup ∪ ( FPPV(o→oBN) ⊕ BPPV(oBN→dBN) ⊕ FPPV(dBN→d) )
if Frag(o) == Frag(d): PO_sup ← PO_sup ∪ FPPV(o→d)   # within-fragment
return DominanceCheck(PO_sup)

MBOR-Adv: 2D cost-interval and multi-edge pruning (Alg 4–5)

The combination above is the bottleneck because many boundary-node pairs cannot contribute. MBOR-Adv builds a two-dimensional cost-interval (2DCI) for each pair from the segment cost-sets in constant time, selects the most lower-left "benchmark" 2DCI, and prunes any pair whose 2DCI is dominated before doing its Minkowski sum. A second technique prunes multi-edges during encoding using a bi-objective triangle inequality and reverse-order edge enumeration, reducing the boundary multigraph's encoding cost.

7. Two-dimensional cost-interval pruning

For a boundary-node pair, the combined route's best conceivable cost is its ideal corner (min total c1, min total c2), and its whole frontier lies inside a rectangle anchored there. If that corner is already dominated by the frontier found so far, every path through the pair is dominated, so the pair is skipped. Processing pairs in ideal-corner order tightens the frontier early and prunes the rest, all while returning the identical exact answer.

Figure 4 (redrawn). Each boundary-node pair maps to a 2D cost-interval (rectangle) in objective space. A pair whose ideal corner is dominated by the benchmark contributes nothing and is pruned before the expensive Minkowski combination.
Correctness & completeness (Theorem 4.1). At termination MBOR returns a set that is both a subset and a superset of the true Pareto-optimal set, i.e. exactly the complete Pareto frontier. MBOR-Basic and MBOR-Adv return identical results; Adv is only faster.

9. Independent reproduction (Rust)

A from-scratch Rust reimplementation (arunshar/mbor, hub at arunshar.com/mbor) reproduces the method on the open DIMACS Bay Area subnetworks. Every answer is checked against two exact baselines (BOA*, bi-objective Dijkstra) and an exact full-graph search; on real BAY queries all methods return the identical Pareto frontier (0 mismatches).
