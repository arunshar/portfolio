Abstract

An electric vehicle does not have one best route; it has a trade-off curve. The fastest route may drain the battery; the most energy-efficient may be slow. The right object to compute is the Pareto-optimal set of non-dominated routes, every route for which you cannot improve travel time without spending more energy and vice versa. Computing that set is expensive: bi-objective shortest path is NP-hard in the worst case, with exponentially many Pareto-optimal paths, and a city-scale graph recomputed per query is far too slow for real-time use. MBOR makes it interactive by precomputing and reusing the Pareto structure with three data structures: a boundary multigraph that summarizes inter-region path trade-offs, a multi-level encoded Pareto frontier view that stores the non-dominated sets compactly, and 2D cost-interval pruning with multi-edge pruning that discards dominated path fragments early.

Honest scope. MBOR is preliminary research (algorithm design plus a reported timing result), developed with Prof. Shashi Shekhar and tied to Task 4 of his NSF proposal on multi-objective path selection. The reported result, more than 10x faster online than prior methods on real-world road-network data, is from that preliminary work; it is not yet a public, reproducible benchmark, and unlike the other portfolio projects there is no public code repository or live demo here. The paper is in preparation. This page describes the design and the claim's provenance honestly rather than presenting a finished, reproduced artifact.

Why bi-objective routing is hard, and the idea

Single-objective shortest path (Dijkstra, A*, contraction hierarchies) returns one route in microseconds. The moment you have two costs that cannot be collapsed into one number, energy and time, a single route is the wrong answer: different drivers sit at different points on the trade-off. You want the whole non-dominated frontier. The catch is that the frontier can be exponentially large and recomputing it per query does not scale. MBOR's bet is the same one that makes single-objective routing fast at city scale, preprocess once, query fast, lifted to the bi-objective setting.

Preprocess the Pareto structure once over a region-partitioned boundary multigraph; at query time, prune dominated fragments and return the full non-dominated set fast.

The three data structures

Boundary multigraph

Partition the road network into regions and summarize, for each pair of boundary nodes, the Pareto-optimal inter-region trade-offs as parallel (multi) edges. Queries hop region-to-region over this small graph instead of the full network, the hierarchical-acceleration idea (Shekhar's BMR) lifted to two objectives.

Multi-level encoded Pareto frontier view

Store the non-dominated cost sets compactly and at multiple levels of granularity, so the precomputed frontier can be retrieved and stitched without re-deriving it per query.

2D cost-interval + multi-edge pruning

At query time, bound each partial path by a 2D (energy, time) cost interval and discard fragments that are already dominated, plus collapse redundant parallel edges, so the search never enumerates the exponential frontier in full.
