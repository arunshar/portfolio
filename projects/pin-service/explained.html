Pin-Service (PUDO pin selection) — Explained

One line: the production serving path for choosing where a vehicle should actually stop, candidate generation → hard-constraint filtering → ML scoring → congestion-aware re-ranking → load shedding, behind a gRPC API with full observability.

Honest status. This is a clean, tested reference implementation (17 files, ~1,100 LOC; unit tests for each stage + a Locust load test) of the canonical serving path. The ML scorer is a gradient-boosted regressor trained on a synthetic satisfaction label; the HD-map and supply signals are fixtures/stubs. So the engineering pattern is real and runnable; the model and data are illustrative.

1. The problem and why it's not trivial

You can't just pick "the lat/lon nearest the rider/door." The stop has to be (a) on a drivable lane, (b) legal to stop at (no bus stops, hydrants, no-stop zones), (c) walkable from the actual destination, and (d) globally coordinated so demand spikes (a stadium exit, an apartment complex) don't all get assigned the same curb. That last point is what turns a lookup into a systems problem.

2. Architecture and data flow

 request (rider/destination) ──► gRPC server (pin_service/server.py)
        │
        ▼
 1. generate_candidates       → nearby candidate pins (candidate_gen.py)
        ▼
 2. filter_feasible           → drop illegal/unreachable pins (constraints.py)  [HARD GATE]
        ▼
 3. score_candidates          → gradient-boosted "satisfaction" score (scorer.py)
        ▼
 4. congestion re-rank        → subtract a penalty from over-assigned cells (congestion.py)
        ▼
 5. load shed                 → if global rate too high, fail fast (congestion.py)
        ▼
 best pin  +  OpenTelemetry spans (telemetry.py)

The ordering is the whole point: hard constraints before ML, ML before global coordination, coordination before load shedding.

3. Module-by-module walkthrough

pin_service/candidate_gen.py — generate_candidates(...) returns nearby PinCandidates (lat/lon + lane + metadata) around the requested location. The cheap, high-recall first stage: cast a wide net, let later stages prune.

pin_service/constraints.py — the hard gate. HDMapView snapshots HD-map polygons (drivable_area, no_stop_zones); is_feasible(candidate, hd_map) returns a strict boolean (on a drivable lane AND not in a no-stop zone), and filter_feasible keeps only the survivors. The doc comment states the invariant explicitly: ML scoring NEVER overrides a hard-constraint failure. Legality and reachability are not negotiable by a learned score.

pin_service/scorer.py — score_candidates(...) runs each surviving candidate through a gradient-boosted regressor (joblib, loaded once at import as a lazy singleton so gRPC worker threads share it). Features include historical success per H3 cell and local supply. The label is a synthetic "rider satisfaction" in training. There are parity tests so a Python-vs-C++ reimplementation would match, the kind of thing production serving needs.

pin_service/congestion.py — CongestionTracker does two jobs. Re-ranking: when a cell's assignments-in-window exceed threshold, subtract a penalty from its candidates' scores so the re-ranker prefers a cooler nearby cell (spreads load across curbs). Load shedding: if the global assignment rate exceeds load_shed_ratio × threshold, the service fails requests fast rather than melting down, the demand-protection valve.

pin_service/server.py — the gRPC service wiring the five stages, plus telemetry.py (OpenTelemetry spans) and metrics.py for observability. client/ has a sample client; load_tests/locustfile.py drives synthetic load.

Tests: test_candidate_gen, test_constraints, test_scorer, test_congestion — one per stage, asserting the contract of each (constraints are strict, congestion penalizes hot cells, scorer parity).

4. The three ideas worth reciting

  1. Constraints before ML (and ML can't override them). Safety/legality are hard booleans evaluated before scoring; a high model score never rescues an illegal pin. This is the single most important production-ML principle: keep the learned component inside a hard-constraint envelope.
  2. Global coordination via congestion penalties. Local greedy scoring would pile everyone onto the best curb. A per-cell congestion penalty turns independent requests into a load-balanced fleet decision without a central optimizer.
  3. Graceful degradation via load shedding. Under a demand spike, failing some requests fast keeps latency bounded for the rest, far better than uniformly slow or a cascading collapse.

5. What I'd do next

  1. Train the scorer on a realistic label (e.g. actual walk distance + dwell + reassignment rate) instead of the synthetic satisfaction stub.
  2. Replace the HD-map fixture with real map polygons (OSM lanes + curb data) so feasibility is genuine.
  3. A/B the congestion penalty to show it actually lowers max-curb load vs greedy.
  4. Tie it to LastMile-SafeSeg: feed the per-pin safety score (curb height, stairs, blocked approach) in as a feasibility/score input, that connects the perception side to the serving side into one stop-selection story.
