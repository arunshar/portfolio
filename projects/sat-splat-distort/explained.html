Sat-Splat-Distort — Explained

One line: make 3D Gaussian Splatting work for non-pinhole cameras (satellite, pushbroom, fisheye, 360) by replacing the one wrong matrix in the standard rasterizer with an exact, per-camera, closed-form Jacobian, plus a small learned grid that mops up residual lens noise.

Honest status up front. The genuine, defensible contribution that is real and tested is the four closed-form camera Jacobians, each numerically validated against torch.autograd (7/7 tests pass), and the learned distortion-prior grid module (runs on CPU). The full GPU 3DGS fit relies on a forked CUDA rasterizer that is loaded lazily and stubbed for CPU smoke, and the benchmark numbers in the README (PSNR 24.2 etc.) are labelled "leaderboard targets," not reproduced results. Repo carries a model-checkpoint-scaffold badge. In short: "the camera math is done and validated; the trained scene checkpoints and the benchmark numbers are the next step."

1. What it is and the gap it fills

3D Gaussian Splatting (3DGS) represents a scene as millions of tiny 3D Gaussian blobs (each with a position, a 3D covariance = shape/orientation, an opacity, and a view-dependent color). To render, you project every 3D Gaussian to the 2D image and "splat" it (draw it as a fuzzy 2D ellipse), then alpha-blend front-to-back. It is fast and photorealistic, and it won a best-paper at SIGGRAPH 2023.

The catch: 3DGS was derived for the perspective pinhole camera, where projection is (x, y, z) -> (fx·x/z, fy·y/z). Projecting a 3D covariance (not just a point) to 2D uses a linear approximation called the EWA (Elliptical Weighted Average) splat, which needs the Jacobian of the projection (the 2×3 matrix of partial derivatives). For a pinhole camera that Jacobian is a simple affine form baked into the CUDA rasterizer.

Real geospatial / immersive cameras are not pinhole:

  • Satellite imagery ships as an RPC (Rational Polynomial Coefficient) model: pixel = ratio of two cubic polynomials in (lon, lat, height).
  • Pushbroom scanners (Sentinel-2, Landsat, SPOT) build the image one row at a time as the satellite flies, so the along-track axis is a time index, not a projective one.
  • Fisheye lenses use r = f·θ (angle), not r = f·tan θ.
  • 360 panoramas map directions to longitude/latitude (equirectangular).

For all of these, the pinhole EWA Jacobian is wrong, so vanilla 3DGS smears Gaussians incorrectly, worst at the image periphery. Existing fixes either undistort the image first (throws away peripheral detail) or hand-roll a per-camera approximation. Sat-Splat-Distort's idea: keep one unified rasterizer but swap in the exact analytic Jacobian for whichever camera you have, dispatched by a flag, and add a tiny learned correction for whatever the analytic model misses.

2. Architecture and data flow

                          ┌─────────────────────────────────────┐
   3D Gaussian field      │  cameras/{rpc,pushbroom,equidist,    │
   (means, scales,        │           equirect}.py               │
    quats, opacity, SH)   │   project(X) -> (u,v)                │
        │                 │   jacobian(X) -> 2x3  ◄── the IP     │
        ▼                 └─────────────────────────────────────┘
  models/splat_pipeline.py            │
  SatSplatPipeline.render():          │ uv, J
   1. uv = camera.project(means)      ▼
   2. J  = camera.jacobian(means) ─────────────►  CUDA rasterizer fork
   3. dΣ = distortion_grid(uv)  ──────────────►  computeCov2D(J, dΣ): Σ2D = J W Σ3D Wᵀ Jᵀ + dΣ
   4. rasterize -> image                          alpha-blend -> pixels

The only thing different from stock 3DGS is steps 2–3: instead of the affine pinhole Jacobian, we pass the camera-specific analytic J, plus a learned 2×2 covariance bump dΣ.

3. Module-by-module walkthrough

src/sat_splat/cameras/base.py — the interface. CameraModel is an abstract class with two methods every camera must implement: project(X) (3D point → 2D pixel) and jacobian(X) (the analytic 2×3 derivative). It also ships autograd_jacobian(X), which computes the same Jacobian by automatic differentiation (looping over the 2 output components and calling torch.autograd.grad). That reference is what the tests compare against; it is the safety net that proves the hand-derived calculus is correct.

src/sat_splat/cameras/rpc.py (229 LOC, the most involved). Implements the satellite RPC model.

  • RPCMeta dataclass holds the 90 numbers an RPC file carries: 4 offsets + 4 scales for (line, sample, lat, lon, height) and four 20-coefficient cubic polynomials (line numerator/denominator, sample numerator/denominator).
  • _monomials(L,P,H) builds the 20-term cubic basis [1, L, P, H, LP, LH, PH, L², …, H³] in the exact NGA standard order (RPC_MONO_ORDER).
  • _monomial_grads returns the analytic derivatives of all 20 monomials w.r.t. L, P, H (hand-written, e.g. d/dL of L²P = 2LP).
  • project normalizes world (lon, lat, h) to internal (L, P, H), evaluates line = (Σ mono·line_num)/(Σ mono·line_den), same for sample, then denormalizes to pixels.
  • jacobian applies the quotient rule (N'D − ND')/D² to each polynomial ratio, then chains through the normalization (divide by lon/lat/height scales) and the pixel denormalization (multiply by sample/line scales).

src/sat_splat/cameras/pushbroom.py — linearized single-line scanner (Gupta & Hartley 1997). u = fu·(X·a)/(X·c) (cross-track, pinhole within a row), v = fv·(X·b) (along-track, linear in scan time). The Jacobian is du/dX = fu·(a/(X·c) − (X·a)·c/(X·c)²) and dv/dX = fv·b (constant). The "v is linear, not projective" is exactly why stock 3DGS is wrong here.

src/sat_splat/cameras/equidist.py — equidistant fisheye, r = f·θ. Projects via θ = acos(z/‖X‖), radial scale s = f·θ/ρ where ρ = √(x²+y²). The Jacobian uses the identities dθ/dx = xz/(r²ρ), dθ/dy = yz/(r²ρ), dθ/dz = −ρ/r², then a product rule on s·x and s·y. The doc comment spells out the 1/ρ³ residual term, the trickiest part.

src/sat_splat/cameras/equirect.py — equirectangular 360. φ = atan2(x,z) (longitude → u), θ = asin(y/r) (latitude → v). Closed-form Jacobian: dφ/dx = z/ρ², dφ/dz = −x/ρ² (with ρ²=x²+z²), dθ/dx = −xy/(r²ρ), dθ/dy = ρ/r², dθ/dz = −zy/(r²ρ), scaled by W/2π and −H/π.

src/sat_splat/models/distortion_grid.py — DistortionPriorGrid, a learnable 64×64×16 grid of latent codes laid over the image. For a Gaussian landing at pixel (u,v): bilinearly sample a 16-d token (F.grid_sample), push it through a 3-layer MLP, output a symmetric positive-semidefinite 2×2 perturbation [[a,b],[b,c]] (softplus on the diagonal to keep it PSD, off-diagonal free). It is initialized near zero (last layer zeroed, eps_init=1e-3) so the analytic Jacobian dominates early training and the grid only corrects residuals (lens artifacts, RPC fit noise, atmospheric refraction). regularization() is an L2 on the tokens encouraging a smooth, low-frequency distortion field.

src/sat_splat/models/splat_pipeline.py — SatSplatPipeline(nn.Module). Holds the per-scene Gaussian parameters (means, scales, quats, opacities, colors as SH coefficients) + the distortion grid + a camera. render() runs project → jacobian → distortion → CUDA rasterize. _lazy_rasterizer() imports the forked CUDA kernel only on first render (so the package imports fine CPU-only). from_pretrained / save_pretrained give a HuggingFace-style load/save (scene.pt + distortion_grid.pt).

tests/test_cameras.py — the validation that is the contribution. For each camera it samples random 3D points in a sensible region, computes the analytic Jacobian and the autograd Jacobian, and asserts they match elementwise (tolerances 1e-4 to 5e-2). RPC is tested with a toy well-conditioned RPC and with a full nontrivial cubic to exercise every monomial. If any of these fail, the rasterizer would render garbage — so passing them is the proof the math is right.

Other dirs: data/{dfc2019,matterport360}.py (dataset loaders), eval/dfc2019_bench.py (benchmark harness), training/train.py (Hydra-configured fit loop), space/app.py (Gradio demo), cuda_rasterizer/ (the INRIA fork).

4. The core math, from scratch

Why a Jacobian at all? A 3D Gaussian has a covariance Σ3D (a 3×3 matrix describing its ellipsoid shape). To draw it on screen you need its 2D shadow Σ2D. Projection π: R³→R² is nonlinear, so you linearize it at the Gaussian's center with the Jacobian J = dπ/dX (2×3). Then the standard EWA result is

Σ2D = J · W · Σ3D · Wᵀ · Jᵀ

where W is the world→camera rotation. Everything hinges on J being correct for your camera. For a pinhole, J is the familiar [[f/z, 0, −fx/z²],[0, f/z, −fy/z²]]. The whole project is: derive and validate the right J for four other cameras, and plug it into the same rasterizer.

Worked example — the pushbroom Jacobian (the cleanest one to recite): Projection u = fu·(X·a)/(X·c), v = fv·(X·b). Differentiate u with the quotient rule treating a, c as constant vectors: du/dX = fu·[ a·(X·c) − (X·a)·c ] / (X·c)² = fu·( a/(X·c) − (X·a)c/(X·c)² ). And v = fv·(X·b) is linear, so dv/dX = fv·b (a constant). That constant along-track row is the signature of a pushbroom and the reason the projective pinhole Jacobian is structurally wrong.

The distortion grid then adds a learned dΣ so Σ2D = J W Σ3D Wᵀ Jᵀ + dΣ(u,v), capturing what no closed form can (manufacturing tolerances, refraction). It starts at ~0 so it never fights the analytic term early on.

6. Tests, datasets, benchmarks

  • Tests: test_cameras.py (7 Jacobian-vs-autograd) + test_smoke.py (13 import/forward/UI checks). 20 total, CPU, <15 s.
  • Datasets (targeted): IEEE GRSS DFC2019 Track-3 (satellite multi-view, the standard satellite-NVS benchmark), Matterport3D 360, KITTI-360 fisheye.
  • Baselines cited (from their papers, not re-run): Sat-NeRF, EO-NeRF, plain 3DGS. The "ours 24.2 PSNR" row is an aspirational target.

7. What I'd do next (to turn the scaffold into a result)

  1. Build the CUDA fork and fit one real scene. A single DFC2019 AOI rendered with the analytic RPC Jacobian, with vs without the distortion grid, gives one honest qualitative + PSNR number. This is the single highest-value step.
  2. Ablation that sells the idea: render the same scene with the pinhole Jacobian vs the analytic Jacobian and show the periphery error map. That figure is the paper.
  3. gsplat instead of the INRIA fork. The gsplat library exposes a Python computeCov2D hook, so you could inject J and dΣ without maintaining a CUDA fork at all — much easier to run and defend.
  4. Quantify the distortion grid's marginal value (turn it off, measure the PSNR delta) so you can say exactly what the learned part buys over the closed form.
  5. One synthetic exactness test: a scene where ground-truth depth is known, to show the analytic Jacobian recovers geometry the pinhole one cannot.
